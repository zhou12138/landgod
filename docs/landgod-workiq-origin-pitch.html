Page 1 · 真实问题与最初方案

WorkIQ 没有 API，Societas 怎么调用用户电脑上的 CLI？

这是云上 Agent 落地时最典型的“最后一公里”问题：Agent 能理解任务，但真正能完成任务的工具、登录态和网络环境，在用户自己的工作设备上。
LandGod
从 WorkIQ Bridge 开始

业务目标

让 Societas 集成 WorkIQ，帮助用户查询状态、执行流程、生成总结，而不是让用户手动复制 CLI 输出。

Societas 在云上，负责理解目标、规划任务、持续推理
WorkIQ 能力在用户工作电脑上的 workiq CLI
用户期望“Agent 直接帮我完成”，不是再让我手动当中转站
这不是简单集成一个 API，而是让云上 Agent 接触真实工作环境的问题。

不可绕过的约束

WorkIQ Case 暴露了 Local Computer Access 背后的真正难点。

WorkIQ 当时没有标准 API，只有本地 CLI
CLI 依赖本地登录态、配置、公司网络和用户设备环境
个人设备不能外露端口：NAT、防火墙、公司安全策略都不允许
AAD token、cookie、公司环境不能直接搬到云上 Agent
Agent 有脑子，但缺少一个能安全接触本地工具环境的执行 Harness。

最初方案：后端桥 + 本地 Worker

用户工作设备主动连接云端 Gateway。Societas 只发任务意图，Gateway 路由到本地 Worker，由 Worker 调用本机 WorkIQ CLI。

第一性方案：不是让个人电脑暴露公网端口，而是让本地 Worker 主动连接云上 Harness。
第一性问题：云上 Agent 要真正完成工作，必须有一个安全、可运维的执行 Harness，连接用户本地工具环境。

Page 3 · 真实工作流：注册能力与调用能力

Agent 如何一步一步调用 Worker 的本地能力？

LandGod 的真实链路分两段：Worker 先通过 WebSocket 注册身份、包装并发布工具能力；Agent 后续通过 Skill/配置知道 Gateway，再用 HTTP /tool_call 触发一次受签名、可路由、可审计的本地执行。
Real Workflow
Register · Publish · Call

A. Agent 发现入口，Worker 注册并发布能力

上方是 Agent 如何知道 LandGod/MCPHub；下方是 Worker 如何把本地内置 tools 和 MCP tools 包装并发布给 Gateway。

真实点：Agent 通过 Skill/config 知道 Gateway；Worker 把内置工具 + MCP tools 包装进 toolRegistry 后，通过 update_tools 发布能力目录。

B. Agent 发起 tool_call 后的真实路径

以我作为 Agent 调用某个 Worker 的能力为例：我通过 Skill/配置知道 Gateway，然后用 HTTP API 进入这条链。

入口：Agent 通过 LandGod Skill / 配置知道 Gateway 地址和 /tool_call API。
路由：Gateway 按 connection_id、clientName、labels 查找在线 Worker。
执行：Worker 校验签名和 tool binding 后调用 toolRegistry.callTool。
真实端到端协议：Agent 通过 Skill/配置知道 LandGod → HTTP(S) 调 Gateway；Worker 主动用 WS(S) 注册并 update_tools；Gateway 将 HTTP /tool_call 转成 WS tool_call 并返回结果。轻量 Harness：Gateway governs, Worker executes, Activity proves.
