Page 1 · 真实问题与最初方案

WorkIQ 没有 API，Societas 怎么调用用户电脑上的 CLI？

这是云上 Agent 落地时最典型的“最后一公里”问题：Agent 能理解任务，但真正能完成任务的工具、登录态和网络环境，在用户自己的工作设备上。
LandGod
从 WorkIQ Bridge 开始

业务目标

让 Societas 集成 WorkIQ，帮助用户查询状态、执行流程、生成总结，而不是让用户手动复制 CLI 输出。

Societas 在云上，负责理解目标、规划任务、持续推理
WorkIQ 能力在用户工作电脑上的 workiq CLI
用户期望“Agent 直接帮我完成”，不是再让我手动当中转站
这不是简单集成一个 API，而是让云上 Agent 接触真实工作环境的问题。

不可绕过的约束

WorkIQ Case 暴露了 Local Computer Access 背后的真正难点。

WorkIQ 当时没有标准 API，只有本地 CLI
CLI 依赖本地登录态、配置、公司网络和用户设备环境
个人设备不能外露端口：NAT、防火墙、公司安全策略都不允许
AAD token、cookie、公司环境不能直接搬到云上 Agent
Agent 有脑子，但缺少一个能安全接触本地工具环境的执行 Harness。

最初方案：后端桥 + 本地 Worker

用户工作设备主动连接云端 Gateway。Societas 只发任务意图，Gateway 路由到本地 Worker，由 Worker 调用本机 WorkIQ CLI。

第一性方案：不是让个人电脑暴露公网端口，而是让本地 Worker 主动连接云上 Harness。
第一性问题：云上 Agent 要真正完成工作，必须有一个安全、可运维的执行 Harness，连接用户本地工具环境。

Page 2 · 个人身份、权限与多设备

这不是单台电脑访问，而是个人身份下的多设备工具入口

Societas 接入 AAD 鉴权 token 后，LandGod / MCPHub Gateway 可以把“这个用户是谁、能访问什么、哪台设备有能力”串起来，形成个人版 Execution Harness。
MCPHub Gateway
Identity · Devices · Tools

个人用户维度

一个用户，多个工作现场。

用户不是只有一台电脑，而是拥有一组带不同登录态、网络和工具能力的个人设备。

AAD token 证明 Societas 当前代表哪个用户发起请求
Gateway 以用户身份为边界，管理设备、能力和授权范围
不同设备可暴露不同 capability：WorkIQ、Azure CLI、repo、Portal、DevBox
用户价值：Agent 不只是访问“这台电脑”，而是使用“我的工作环境”。

权限与操作边界

身份决定范围，能力决定路由。

LandGod 不把设备简单暴露给 Agent，而是把可执行能力注册成受控工具。

身份：AAD token / agent_id 绑定请求来源和用户上下文
权限：哪些 Agent、哪些工具、哪些设备、哪些 credential_ref 可以使用
操作：读文件、跑 CLI、访问 Portal、调用 MCP 工具都变成显式 capability
可见性：Activity History 记录谁在什么设备上调用了什么能力
Personal Scope围绕单个用户的设备和权限组织
Safe Actions每次执行都有身份、路由和能力边界
个人维度：Local Computer Access 不是裸访问电脑，而是以用户身份组织多设备、多能力和可控操作。

Page 3 · 技术实现与模块

LandGod / MCPHub 如何实现这个 Harness？

源码里的实现不是一个裸 WebSocket 转发器，而是“Gateway 控制面 + Worker 执行面 + 签名鉴权 + 凭据授权 + Activity 可见性”的组合，形成 Intent → Govern → Execute → Observe 的闭环。
Execution Harness
Gateway · Worker · Trust

1. Gateway 控制面

云上 Agent 不直接打到个人设备，而是调用 Gateway 的 HTTP API。Gateway 是 Harness 的治理面：负责路由、授权、签名和可见性。

HTTP API：/tool_call、/batch_tool_call、/agents/heartbeat、/tasks、/logs/tool-calls
设备路由：按 clientName、connection_id、labels 选择目标 Worker
任务管理：同步执行、异步任务、队列、超时、traceId / taskId / requestId
控制策略：Agent / Worker / Tool 可 enable / disable，WebUI 展示 Activity
Gateway 是 Harness 的控制面：不保存用户工作环境，但负责把意图安全送到正确设备。

2. Worker 执行面

Worker 是 Harness 的执行面，跑在用户设备上，主动向 Gateway 建立 WebSocket 出站连接，因此个人设备不需要暴露端口。

注册握手：Worker 用 token 连接 Gateway，绑定 clientId / clientName / labels / session
本地执行：Worker 验证请求后调用本机工具，并把结果回传给 Gateway
能力边界：permission profile、approval mode、工具白名单限制本地可执行范围

3. 鉴权、签名与凭据

Harness 的关键不是“能远程执行”，而是每一次执行都绑定设备、会话、工具、参数和短期授权。

Worker Token：Worker 连接时通过 Bearer token 鉴权，可用服务端 token binding 绑定设备身份和 labels
签名 tool_call：Gateway 用 Ed25519 签名 requestId、session、connection、tool、arguments hash、iat/exp、nonce
Worker 校验：校验 user/client/connection/session/keyId、过期时间、body hash、签名和 replay nonce
Credential Broker：Agent 只传 credential_ref，Gateway 签发短期单次 grant，Worker 交换后本地注入 secret
Activity / Audit：traceId 串起 received、dispatch、grant、exchange、result、timeout、blocked
技术结论：LandGod 是 Agent Execution Harness 的个人实现，不是裸远控；安全边界在 Gateway + Worker 双侧共同完成。
Harness 闭环：Agent 发意图，Gateway 做治理和签名，Worker 在本地验证并执行，结果与 Activity 回到云上。

Page 4 · 能力注册与调用工作流

Worker 如何暴露本地能力？Agent 如何路由并调用？

LandGod 的工作流是：Worker 注册本地能力，Gateway 维护能力目录；Agent 发送 query / tool_call，Gateway 根据身份、权限、设备标签和能力匹配路由到具体 Worker。
Workflow
Register · Route · Execute

Worker 能力注册

Worker 启动后主动连接 Gateway，并把“我是谁、我在哪台设备、我能做什么”注册成可路由能力。

身份绑定：token → clientId / clientName / labels / user scope
设备画像：desktop、laptop、cloud-vm、devbox、office-pc 等标签
能力目录：内置工具 + MCP tools + 本地 CLI wrapper + 文件/浏览器能力
权限边界：permission profile、approval mode、tool allowlist、credential policy
Worker 不是暴露整台电脑，而是把本地能力声明成可被授权调用的 capability。

Query → Route → Execute

Agent 不关心工具在哪台机器上，只表达意图和约束；Gateway 负责把请求变成一次受控的本地能力调用。

工作流闭环：Register capability → Match query → Route worker → Sign request → Execute local tool → Return result。
机制总结：LandGod 把个人设备上的本地能力变成可发现、可授权、可路由、可验证执行的 Agent 工具。
