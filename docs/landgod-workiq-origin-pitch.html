Page 1 · 真实问题与架构取舍

WorkIQ 没有 API，Societas 怎么调用用户电脑上的 CLI？

这是一个很典型的“云上 Agent 最后一公里”问题：业务想让 Agent 自动完成工作，但关键工具只存在于用户个人设备里。
LandGod
从 WorkIQ Bridge 开始

业务目标

让 Societas 集成 WorkIQ，帮助用户查询状态、执行流程、生成总结，而不是让用户手动复制 CLI 输出。

Societas 在云上，负责理解目标和持续推理
WorkIQ 能力在用户工作电脑上的 workiq CLI
用户期望的是“Agent 直接帮我完成”，不是再让我手动当中转站
领导视角：这是 AI 产品落地时最常见的断点——Agent 有脑子，但碰不到真实工作环境。

不可绕过的约束

这不是简单发一个 HTTP 请求就能解决的问题。

WorkIQ 当时没有标准 API，只有本地 CLI
CLI 依赖本地登录态、配置、公司网络和用户设备环境
个人设备不能外露端口：NAT、防火墙、公司安全策略都不允许
凭据、token、cookie、公司环境不能直接搬到云上 Agent
专业用户视角：我愿意让 Agent 帮我做事，但不愿意把电脑变成公网服务，也不愿意把凭据交出去。

当时的方案选择

我们比较过前端直连和后端桥接，最后放弃前端，选择 Gateway + Worker 出站连接。

前端直连 Worker体验直观、链路短；但依赖浏览器在线，跨域/本地网络复杂，token 暴露面大，一关页面任务就断。
后端 Gateway 桥接Worker 主动连云端，无需暴露端口；Agent、队列、权限、日志、重试都能后端统一处理。
结论：Societas 是云上 Agent，不能依赖用户打开某个网页；必须有后端桥。
第一性问题：云上 Agent 要真正完成工作，必须有一条安全、可运维的通道连接用户本地工具环境。

Page 2 · 从单点集成到产品机会

后来发现：这不是 WorkIQ 专属问题，而是个人生产力入口

Gateway 出现后，我们发现这座桥可以泛化：让任意云上 Agent 使用用户个人设备上已经配置好的 CLI、文件、浏览器和工作环境。
MCPHub Gateway
个人工具运行时

专业个人用户视角

“让我的 Agent 用我的工具箱。”

高级用户真正的生产力资产，已经沉淀在自己的设备上：登录态、CLI、repo、脚本、VPN、公司 Portal。

用户价值：少复制、少截图、少切机器，Agent 可以直接使用我已经配置好的工作环境。

领导视角

从一次集成，变成可复制的入口。

WorkIQ Case 验证的是一类更大的需求：云上 Agent 要落地，必须连接用户已有的本地工具生态。

可泛化：不只 WorkIQ，所有本地 CLI / Portal / 文件环境都能接入
高频刚需：熟练 Agent 用户每天都会遇到复制、上传、切机器的问题
差异化：不是再做一个 Agent，而是补齐 Agent 的本地执行层
安全边界：凭据留在设备，Gateway 只做授权、路由和可见性
Less manual work少复制、少截图、少手动执行
More real actionsAgent 从建议者变成执行者
最终定位：LandGod / MCPHub Gateway 是个人生产力提效工具，让云上 Agent 玩转用户自己的本地工具箱。
