Origin Story

Societas 要集成 WorkIQ，
但 WorkIQ 只有本地 CLI

我们的 Agent 在云上，WorkIQ 的能力却在用户工作电脑里的 workiq CLI。没有 API、没有云端入口，怎么办？
LandGod
从 WorkIQ Bridge 开始

真实约束

Societas 可以理解需求、规划任务、总结结果，但它无法直接调用用户电脑上的本地 CLI。

WorkIQ 当时没有标准 API，只有本地 workiq CLI
CLI 安装在用户工作电脑上，依赖本地登录态、配置和网络环境
Societas Agent 在云上，拿不到用户电脑上的命令行能力
不能把用户凭据、token、公司环境直接搬到云上
问题不是 Societas 不够聪明，而是它缺少一座通向用户本地工具环境的桥。

最初要解决的问题

我们需要一个机制，让云上的 Societas 可以在用户授权下，把任务送到工作电脑，由本地环境调用 WorkIQ CLI，再把结果回传。

Agent 只表达意图，不直接持有本地凭据
用户电脑负责执行 workiq ...
结果回到云上 Agent，用于继续推理和生成报告
执行边界由用户控制，可见、可撤销
这就是 LandGod 的起点：不是先做平台，而是先搭一座让 Agent 够到本地工具的桥。
起点：Societas 在云上；WorkIQ 在用户电脑上；LandGod 负责把两者安全连接起来。

From Bridge to Gateway

后来发现：这座桥可以泛化

一开始只是“工作电脑直接连云上 Agent”。加入 Gateway 后，LandGod 从 WorkIQ 专用桥，演进成可泛化的 MCPHub Gateway。
MCPHub Gateway
通向本地工具生态

第一阶段：点对点桥

Societas ↔ 用户工作电脑 ↔ WorkIQ CLI

最初实现没有 Gateway。目标很直接：让用户工作电脑连接云上，让 Societas 可以触发本地 WorkIQ CLI。

解决了 WorkIQ 无 API 的问题
但还是偏一次性连接，难以管理多 Agent、多设备、多工具

第二阶段：MCPHub Gateway

任意 Agent → Gateway → 任意设备 Worker → 任意本地工具

加入 Gateway 后，我们发现这不是 WorkIQ 专属问题，而是云上 Agent 访问本地工具生态的通用问题。

从 WorkIQ 专用桥，泛化成云上 Agent 的本地工具 Gateway
可以接任意 Agent、任意用户设备、任意本地 CLI / Portal / 文件环境
抽象后的 LandGod：不是只集成 WorkIQ，而是让云上 Agent 通过 MCPHub Gateway 使用用户设备上的整个本地工具生态。
