Page 1 · 问题与当时方案

Societas 要集成 WorkIQ，
但 WorkIQ 只有本地 CLI

Agent 在云上，工具在用户工作电脑上。没有 API，只有本地 workiq CLI，这就是最初必须解决的“桥”的问题。
LandGod
WorkIQ Bridge 起点

遇到的问题

Societas 可以理解需求、规划任务、总结结果，但它无法直接调用用户电脑上的 WorkIQ CLI。

WorkIQ 当时没有标准 API，只有本地 workiq CLI
CLI 安装在用户工作电脑上，依赖本地登录态、配置和公司网络
Societas Agent 在云上，天然访问不到这台机器上的工具
不能把用户凭据、token、公司环境直接搬到云上
瓶颈不是 Agent 不够聪明，而是它碰不到用户本地工具箱。

当时的方案：先搭一座桥

让用户工作电脑主动连接云上，云上的 Societas 只发送任务，本地 Worker 调用 WorkIQ CLI，再把结果回传。

跑通 WorkIQ 无 API 集成
保留本地凭据和环境，不把公司登录态搬到云上
第一性问题：云上 Agent 很强，但用户真正的工具、登录态和执行环境在本地设备上。

Page 2 · 泛化后的产品价值

后来发现：这不是 WorkIQ 专属问题，
而是个人生产力提效工具

加入 Gateway 后，这座桥可以从“WorkIQ 专用连接”泛化成 MCPHub Gateway：让云上 Agent 使用用户个人设备上的整个本地工具生态。
MCPHub Gateway
个人工具运行时

抽象后的洞察

每个高级用户，都有一个本地工具箱。

WorkIQ 只是第一个例子。真正可泛化的是：很多高价值能力都只存在于用户自己的设备上。

workiq
az
copilot
gh
kubectl
ssh
Portal
local repo

产品价值变了

LandGod 不只是 WorkIQ 集成方案，而是让云上 Agent 调用个人设备工具生态的生产力基础设施。

用户不用再复制输出、截图、上传文件、手动跑命令
Agent 可以使用用户已经配置好的 CLI、浏览器和本地 repo
凭据、cookie、SSH key、VPN 环境继续留在用户设备
多台设备都能变成 Agent 的本地工具运行时
云上 Agent 负责思考，MCPHub Gateway 连接设备，本地工具负责执行。
最终定位：LandGod / MCPHub Gateway 是个人生产力提效工具，让云上 Agent 玩转用户自己的本地工具箱。
