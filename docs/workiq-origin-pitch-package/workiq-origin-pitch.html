Problem → Generalization

1. Start with WorkIQ

Societas needed to operate WorkIQ for the user, but the capability existed only on the user’s local computer.

Cloud Agent handles reasoning and task planning
WorkIQ capability lived behind workiq CLI
User expects completed actions, not manual relay
The WorkIQ gap reveals the product category: secure local execution for Agents.

2. The Boundary is General

WorkIQ is one example of a broader pattern: valuable tools, identity, and work context remain local.

No public API; local CLI only
AAD tokens, cookies, VPN, device config stay local
No inbound exposure on personal devices
Worker connects outbound; Gateway routes intent
The same boundary appears across CLI tools, browsers, repos, VPNs, and internal systems.

3. Direct Generalization

The product is not a WorkIQ bridge; it is Local Computer Access for cloud Agents.

Generalization: turn personal devices into governed local execution runtimes for cloud Agents.
Thesis: The WorkIQ problem directly generalizes into Local Computer Access: cloud intelligence, local execution.

03 · Product Workflow: Register, Invoke, Execute
Product Flow
Register · Invoke · Return

A. MCPHub Client Connects + Publishes Tools

Client-initiated outbound connection keeps personal devices private while publishing a user-scoped capability catalog.

Protect: no inbound port on personal devices. Client establishes session + server-key binding, then publishes a user-scoped capability catalog.

B. Agent Calls + Client Executes Locally

Agent calls Gateway over HTTP. Gateway selects the MCPHub Client, signs the request, and the Client executes on the local tool runtime.

Route / Execute: Gateway selects Client by connectionId, clientName, or labels; Client verifies signed meta, then invokes toolRegistry.callTool.
Product Flow: Clients publish capabilities first; Agents invoke later through Gateway; results return through Gateway with Activity trace.
